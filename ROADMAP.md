# План работы команды MusicTech

**Цель:** научная статья на тему *real-time score following* с
HMM/OLTW-бейзлайном и инновационным RL-модулем для подачи на
конференцию **«Центральный Телеграф»** (submission текста — **3 мая
2026**, доклад — 17 мая 2026), пилотный прототип «виртуального
оркестра» для фортепианного концерта (MVP), а в перспективе —
submission на **ISMIR 2026**.

**Команда:** Никита Новицкий (HMM, координация), Никита Борисов
(HMM, ML), плюс ещё 5 человек, по одному на каждый модуль:
OLTW, real-time / входные данные, датасеты партитур, выходной
модуль, корнер-кейсы, CNN.

Документ устроен по этапам. Для каждого этапа указано: что делаем,
кто отвечает, что должно получиться (deliverable), какие материалы
читать, какие инструменты использовать. Тематические подборки
ссылок вынесены в отдельные файлы:

- `article/docs/literature.md` — ключевые статьи (10+) с краткими аннотациями.
- `article/docs/datasets.md` — все датасеты партитур и исполнений, ссылки и
  лицензии.
- `article/docs/competitors.md` — конкуренты (Cadenza Live, MyPianist,
  Antescofo и др.), что у них тестировать.
- `article/docs/tools.md` — Python-библиотеки, DAW, синтезаторы, JUCE.
- `article/docs/hmm-extensions.md` — современные расширения HMM (HSMM,
  Neural HMM, CRF, switching state-space).

## Этап 2. Сбор и подготовка датасетов (параллельно с этапом 1)

**Цель:** иметь надёжный train / val / test для всех моделей,
включая собственный датасет CU-Concerto-2026.

**Что делать:**

1. **Скачать открытые датасеты** (полный список с прямыми
   ссылками — в `article/docs/datasets.md`):
   - **MAESTRO v3** — фортепиано + MIDI, ~200 ч (Magenta).
   - **MAPS** — фортепиано (synth + Disklavier).
   - **MSMD** — выравнивание audio↔sheet image.
   - **ASAP** — выровненные performance + score (KAIST).
   - **Bach10** — полифонический ансамбль с разметкой.
   - **URMP** — мультиинструмент с видео.
   - **MusicNet** — классическая музыка с разметкой.
   - **Saarland Music Data (SMD)** — фортепиано с MIDI.
   - **GiantMIDI-Piano** — большой корпус MIDI.
   - **ATEPP** — аligned transcribed expressive piano performances.
2. **Подготовить синтетический датасет в стиле прелюдий Шопена**:
   - Загрузить MusicXML прелюдий с
     [imslp.org](https://imslp.org/wiki/Préludes,_Op.28_(Chopin,_Frédéric)).
   - Конвертировать MusicXML → MIDI через `music21` или
     `partitura`.
   - Сгенерировать 40 модифицированных исполнений на каждую
     прелюдию: гауссовский шум онсетов, локальные ритардандо,
     случайные пропуски/повторы.
3. **Собрать собственный датасет CU-Concerto-2026**:
   - Привлечь 4–6 пианистов-консерваторцев (через музкол. и
     консерваторию).
   - Программа: первые части концертов Чайковского b-moll и
     Рахманинова c-moll.
   - Запись через Yamaha CLP-735 (MIDI) + конденсаторный
     микрофон (audio).
   - Ручная разметка: границы тактов, ферматы, ритардандо,
     повторы, пропуски, фальшивые ноты. Использовать `Audacity`
     для аудио и `MuseScore` или `Sonic Visualiser` для разметки.
   - Этика: подписать informed consent, хранить аудио локально.
4. **Стандартизовать формат**:
   - Партитура: MusicXML.
   - Исполнение: MIDI + .wav 44.1 kHz.
   - Выравнивание: CSV с парами `(score_event_id, perf_time_ms)`.
   - Метаданные: `dataset.json` с лицензией, источником, разбиением.
5. **Сделать train/val/test split** на уровне произведений
   (не записей), чтобы исключить утечку.

**Результат этапа:**

- скачанные открытые датасеты в `data/external/` (с
  `data/external/README.md`, где для каждого описан источник и
  лицензия);
- `data/synthetic/chopin/` с 40 × 24 = 960 файлами;
- `data/cu-concerto-2026/` с записями + разметкой + согласиями;
- скрипт `scripts/build_splits.py`, который собирает train/val/test.

## Этап 3. Бейзлайн HMM + OLTW (2–3 недели)

**Цель:** работающий гибридный трекер на синтетических данных,
готовый к замене эмиссии и подключению CNN/RL.

**Что делать:**

1. **HMM-модуль** (`src/tracker/hmm/`):
   - Структуры данных: `HMMState`, `HMMTracker`.
   - Полосная матрица переходов из формулы (4) статьи
     с гиперпараметрами `p_stay`, `p_advance`, `p_skip`.
   - Гауссова эмиссия для MIDI: `b_i(o) = N(o; p_i, sigma^2)`.
   - Forward-рекурсия + нормировка
     (см. Приложение А статьи).
   - Viterbi-рекурсия для офлайн-обработки.
   - Поддержка duration-augmented HMM (HSMM-like).
   - Юнит-тесты на маленьких ручных примерах
     (5 нот, 8 наблюдений).
2. **OLTW-модуль** (`src/tracker/oltw/`):
   - Окно $W$, RunCount, локальная стоимость
     `d(s_i, q_j)`.
   - Псевдокод в Приложении Б статьи.
   - Юнит-тесты с тем же маленьким примером.
3. **Общий API трекера**:
   ```python
   class Tracker(ABC):
       def update(self, observation: NoteEvent) -> int:
           """Returns current best estimate of score position."""
       def confidence(self) -> float: ...
       def reset(self, position_hint: int = 0) -> None: ...
   ```
4. **Гибридная схема** (см. §VIII статьи): HMM как основной,
   OLTW активируется при `confidence < theta_conf`.
5. **Бенчмарк на синтетике**: метрики MAE, F1, ReSync на тестовом
   подмножестве синтетического Шопена.

**Используемые библиотеки** (полный список — `article/docs/tools.md`):

- `numpy`, `scipy.special.logsumexp` для численной устойчивости;
- `numba` для ускорения Forward-рекурсии;
- `pretty_midi`, `mido` для работы с MIDI;
- `music21` или `partitura` для парсинга MusicXML;
- `pytest` для тестов.

**Результат этапа:**

- `src/tracker/` с покрытием ≥ 80 %;
- результаты бенчмарка в `experiments/baseline_hmm/results.json`;
- разделы §VI и §VIII статьи дополнены реальными числами
  (заменить заглушки `X`).

## Этап 4. CNN-фронтенд для аудио (параллельно с этапом 3)

**Цель:** обучаемая эмиссионная модель для аудиовхода, заменяющая
гауссову плотность HMM.

**Что делать:**

1. **Извлечение признаков**: log-mel-спектрограмма
   `128 mel × 40 frames` через `librosa.feature.melspectrogram`
   (sr=22050, hop=512, n_mels=128).
2. **Архитектура**: 4× Conv2D(3×3) + ReLU + MaxPool, GAP,
   FC(128) → softmax(88), как описано в §IX статьи.
3. **Обучение** на MAPS + CU-Concerto-2026:
   - Лосс: cross-entropy по 88 клавишам;
   - Optimizer: Adam, lr=3e-4, 24 epoch;
   - Class imbalance: либо weighted CE, либо oversampling
     редких высот.
4. **Интеграция в HMM**: подменить `b_i(o)` на
   `P_CNN(p_i | o)` от свежеобученной сети.
5. **Корнер-кейсы** (`src/tracker/corners/`):
   - Орнаменты: фильтр по плотности onset’ов в 60 мс окне.
   - Октавные ошибки: замена кандидата.
   - Случайные повторы: фильтр < 25 мс.
   - Полный список — таблица §IX статьи.

**Используемые библиотеки:**

- `torch` 2.3 + `torchaudio`;
- `librosa` для спектрограмм;
- `madmom` (опционально) для готовых onset-detectors.

**Результат этапа:**

- модель в `src/tracker/cnn/checkpoints/cnn_v1.pt`;
- `experiments/cnn_emission/results.json`;
- таблица «полифоническая F1 vs гауссова эмиссия» в §IX.

## Этап 5. RL-модуль предсказания темпа (2–3 недели)

**Цель:** опережающий модуль (изюминка статьи), снижающий
эффективную задержку аккомпанемента без потери точности.

**Что делать:**

1. **Среда** (`src/rl/env.py`):
   - Совместимый с `gymnasium` интерфейс.
   - Состояние, действие, награда — в точности как в §X
     статьи и Приложении В.
   - Симулятор солиста с возмущениями (см. Приложение В).
2. **Behavioural cloning** (этап 1 обучения):
   - Извлечение «учительской» траектории темпа из выровненных
     исполнений CU-Concerto-2026.
   - 2-слойный MLP (128, 64), MSE-лосс. ~25 мин на CPU.
3. **PPO fine-tuning** (этап 2):
   - `stable-baselines3` PPO с клиппингом 0.2, GAE λ=0.95.
   - 5×10⁶ шагов среды, ~12 ч на RTX 3060.
   - Логирование в `wandb` или `tensorboard`.
4. **Интеграция с трекером**: запрос политики на каждом тике
   (10 мс), результат идёт в рендерер как темповый коэффициент
   на 200–500 мс вперёд.
5. **Альтернативные направления** (на потом, не для этой статьи,
   но упомянуть в Conclusion):
   - RLHF / preference learning (Christiano et al. 2017);
   - POMDP-формулировка над belief-состоянием HMM.

**Используемые библиотеки:**

- `gymnasium`, `stable-baselines3`, `torch`;
- `wandb` для логов экспериментов.

**Результат этапа:**

- `src/rl/` с обучаемой политикой и обученными чекпойнтами;
- `experiments/rl_anticipation/` с метриками задержки и MAE;
- §X статьи и Приложение В дополнены реальными числами.

## Этап 6. Интеграция, real-time, UI (1–2 недели)

**Цель:** работающий end-to-end прототип, который слушает
MIDI/микрофон и играет оркестр.

**Что делать:**

1. **Входной модуль** (`src/io/`):
   - MIDI: `python-rtmidi` через USB-MIDI клавиатуру.
   - Аудио: `sounddevice` (PortAudio) + WASAPI/ASIO для низкой
     задержки в Windows.
   - Кольцевой буфер для thread-safe обмена.
2. **Очистка** (`src/io/cleaner.py`):
   - Фильтр коротких событий (< 25 мс).
   - Восстановление октавных ошибок относительно текущего
     HMM-состояния.
   - Агрегация audio onset-кандидатов в окне 40 мс.
3. **Выходной модуль** (`src/renderer/`):
   - `FluidSynth` через `pyfluidsynth` для воспроизведения
     SoundFont’ов оркестра.
   - Time-stretching без изменения высоты: фазовый вокодер
     блоками по 512 сэмплов.
   - Тёплый старт буфера на 100 мс перед началом.
4. **Координация модулей** через очереди и таймеры:
   - входной поток → очистка → tracker.update() →
     RL.predict() → renderer.play_chunk();
   - профайлер пишет распределение задержек в лог.
5. **Минимальный UI** (на выбор):
   - **Вариант A (быстрый):** `tkinter` или `PyQt6` desktop.
   - **Вариант B (красивый):** Electron + React (для будущего
     демо на конференции).
   - **Вариант C (профессиональный):** JUCE-приложение на C++,
     если у кого-то в команде есть опыт; даёт минимальную
     задержку.

**Используемые библиотеки:**

- `python-rtmidi`, `sounddevice`, `pyfluidsynth`;
- `pyaudio` или `sounddevice` для микрофонного режима;
- `JUCE` (C++), `juce_audio_processors` — если идём по варианту C;
- `PyQt6` или `customtkinter` для desktop UI.

**Результат этапа:**

- `src/app.py` — единая точка входа, запускающая прототип;
- видеозапись 30-секундного демо (для конференции и для
  внутренней оценки);
- замеры end-to-end задержки на профайлере (целевое значение
  < 20 мс).

## Этап 7. Эксперименты и метрики (1 неделя)

**Цель:** числа в статье + выводы.

**Что делать:**

1. **Запустить три конфигурации** на тестовой части
   CU-Concerto-2026:
   - HMM (гауссова эмиссия, без RL);
   - HMM-CNN (свёрточная эмиссия, без RL);
   - HMM+OLTW+RL (полная схема, изюминка статьи).
2. **Метрики**:
   - MAE (mean absolute error) выравнивания в номерах нот.
   - F1 онсетов с допуском 50 мс.
   - ReSync (доля времени в режиме повторной синхронизации).
   - Эффективная задержка $L_{\text{eff}}$ — медиана сдвига
     рендера относительно onset солиста.
3. **Графики и таблицы**:
   - Зависимость задержки от горизонта предсказания (TikZ-график
     в §XI статьи).
   - Зависимость ошибки выравнивания от $\sigma_\tau$ rubato.
   - Сводная таблица метрик по конфигурациям.
   - Таблица по сценариям исполнения (rubato, пропуски,
     орнаменты, педаль, шум).
4. **Субъективная оценка**: 6 студентов-консерваторцев
   играют концерт с двумя конфигурациями (HMM vs HMM+OLTW+RL),
   слепое тестирование, оценки по пятибалльной шкале.

**Результат этапа:**

- все таблицы и графики в paper/ обновлены реальными цифрами;
- `experiments/final_results.csv` со всеми сырыми данными;
- 6 заполненных анкет субъективной оценки.

## Критерии успеха проекта

1. **Технический критерий:** end-to-end задержка прототипа на
   репертуаре концерта Чайковского ≤ 20 мс при $\sigma_\tau$ до
   10 % rubato.
2. **Научный критерий:** статья принята на «Центральный Телеграф»
   и подана на ISMIR 2026.
3. **Пользовательский критерий:** ≥ 8 из 10 студентов-
   консерваторцев положительно оценивают опыт репетиции с
   виртуальным оркестром.
4. **Командный критерий:** все 7 модулей доведены до состояния
   «работает + документация + тесты», передаваемого следующему
   потоку студентов и/или продуктовой команде Т-Банка.
